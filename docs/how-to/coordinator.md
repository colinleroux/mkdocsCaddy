**Testing Coordinator Features in Docmosis Tornado 2.11.x**

## Overview

Docmosis Tornado **2.11.0** introduced an enhancement to coordinator templates allowing the `<<coordinator:newFile:>>` instruction to accept a **dynamic value based on a template variable**.

Example:

```
<<coordinator:newFile:$myDocument>>
```

This allows coordinated document outputs to be **named dynamically from the data payload**, enabling flexible multi-document bundles generated from a single render request.

This document describes a simple test that demonstrates the feature.

---

## Test Objective

The goal of this test is to confirm that Tornado can:

1. Loop over a data array  
2. Dynamically select templates  
3. Dynamically name output documents  
4. Produce multiple documents  
5. Return them as a **ZIP archive**

Expected output:

```
coord-test.zip
├─ alpha-summary.pdf
├─ beta-summary.pdf
└─ gamma-summary.pdf
```

---

# **Template Structure**

Create the following templates in the **Docmosis template store**.

```
templates/
 ├─ master.docx
 ├─ alpha-summary.docx
 ├─ beta-summary.docx
 └─ gamma-summary.docx
```

---

# **Coordinator Template**

## **master.docx**

```
<<coordinator:>>

<<rs_documents>>

<<$documentName=documentName>>
<<$templateName=templateName>>

<<coordinator:newFile:$documentName>>
<<refLookup:$templateName>>

<<es_>>
```

## 

## **Explanation**

| Field | Purpose |
| ----- | ----- |
| `<<coordinator:>>` | Marks the template as a coordinator template |
| `<<rs_documents>>` | Begins loop over document list |
| `$documentName` | Stores output filename |
| `$templateName` | Stores template to render |
| `<<coordinator:newFile:$documentName>>` | Creates a new coordinated output file |
| `<<refLookup:$templateName>>` | Renders referenced template |
| `<<es_>>` | Ends the repeat section |

# **Render Request** 

Example request:

```
POST http://localhost:8080/api/render
Content-Type: application/json
```

Payload:

```json
{
  "templateName": "master.docx",
  "outputName": "coord-test",
  "outputFormat": "pdf",
  "compressSingleFormat": true,
  "data": {
    "documents": [
      {
        "documentName": "alpha-summary.pdf",
        "templateName": "alpha-summary.docx"
      },
      {
        "documentName": "beta-summary.pdf",
        "templateName": "beta-summary.docx"
      },
      {
        "documentName": "gamma-summary.pdf",
        "templateName": "gamma-summary.docx"
      }
    ]
  }
}
```

---

# **Expected Response**

The HTTP response should indicate that a ZIP archive was produced.

Example headers:

```
Content-Type: application/zip
Content-Disposition: attachment; filename="coord-test.zip"
X-Docmosis-Zip-Created: true
X-Docmosis-PagesRendered: <total pages across rendered outputs>
```

---

# **Expected Output**

The response body will contain a ZIP archive with three PDF files.

```
coord-test.zip
├─ alpha-summary.pdf
├─ beta-summary.pdf
└─ gamma-summary.pdf
```

Each PDF corresponds to the template specified in the input data.

# **Key Feature Demonstrated**

This test confirms the new Tornado capability:

```
<<coordinator:newFile:$variable>>
```

The output filename can now be **derived from a template variable**, enabling:

* data-driven document naming  
* dynamic document bundles  
* flexible multi-document generation workflows

---

# **Coordinator Template Requirements**

Coordinator templates must follow these rules:

### **1\. The template must begin with**

```
<<coordinator:>>
```

### **2\. Coordinated outputs must include referenced templates**

```
<<ref:template.docx>>
```

or

```
<<refLookup:templateName>>
```

### 

### **3\. To return a ZIP response from the REST API**

Use:

```
compressSingleFormat=true
```
